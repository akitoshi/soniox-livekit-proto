# オンライン診療プラットフォーム: 翻訳設計変更 + 文字起こし永続化

## 背景・現在の構成(前提)

多言語オンライン診療プラットフォーム(国際クリニック向け、自費診療インバウンド患者)を開発中。現在のプロトタイプ構成:

- **フロント**: Next.js 15 (App Router) / Vercel
- **認証**: Clerk(医師は認証済み、患者はゲスト参加)
- **DB**: Supabase
- **ビデオ通話**: LiveKit(クライアントSDK、字幕はテキストストリームで参加者間配信)
- **STT+翻訳**: Soniox リアルタイムAPI(モデル `stt-rt-v5`)。各参加者のブラウザから一時APIキーでSoniox WebSocketに直結(クライアント直結アーキテクチャ)。一時キーはNext.js API Routeで発行
- 想定シナリオ: 医師=日本語話者、患者=外国語話者(例: ドイツ語)の1対1診療。互いの発話を相手の言語の字幕で表示

このアーキテクチャ自体(クライアント直結、サーバーエージェント不使用)は維持する。

> 注(kickoff 検証 2026-07-13): このリポの実装は Next.js 16.2.10 / 認証・DB 未導入 / 字幕は LiveKit data channel(text streams ではない)。上記「前提」は本実装リポの想定構成であり、このリポとの差分は docs/prds/one-way-translation.md に記録済み。

## Task 1: 翻訳モードを two_way → 話者ごとの one_way ×2 に変更

### 課題
現在は `translation: { type: "two_way", language_a: "ja", language_b: "de" }` のような双方向設定を想定している。しかしSonioxのtwo_wayは、設定した2言語以外の第三言語(特に英語)が話された場合、**文字起こしのみで翻訳されない**仕様(公式docs明記)。国際診療では患者・医師とも英語を混ぜて話すケースが頻出するため、これは許容できない。

### 変更内容
各参加者が自分のマイク音声を自分のSonioxセッションに送っている構成を活かし、**one_wayを話者ごとに設定**する:

- 医師側セッション: `translation: { type: "one_way", target_language: "<患者の言語>" }`
- 患者側セッション: `translation: { type: "one_way", target_language: "ja" }`

one_wayは「話されたすべての言語を単一ターゲット言語に翻訳」するため、途中で英語に切り替わっても(mid-sentence code-switching含む)常に受信者の言語へ翻訳される。

- 患者の言語は診療予約時 or 入室時に選択させ、セッション設定に反映する
- **患者言語の選択肢は特定言語をハードコードしない**。Sonioxの対応言語(60+言語)を前提に、対応言語リストを定数ファイル or 設定として一元管理し、そこから選択UIを生成する。想定患者は世界中からの旅行客であり、中国語(簡体/繁体=台湾)、韓国語、ベトナム語などアジア圏からヨーロッパ言語全般まで幅広く来る。日本語(医師側)と英語(共通語フォールバック)だけがファーストクラスで、それ以外は「Sonioxが対応していれば選べる」構造にする
  - 中国語の簡体/繁体など変種の言語コードはSonioxの公式Supported languagesを確認して正確に実装すること(推測でコードを書かない)
  - 選択UIは患者向けなので多言語表示(最低限、各言語を自国語表記で表示: 「中文(简体)」「Tiếng Việt」など)
- `language_hints` には `["ja", "en", <患者の選択言語>]` を動的に渡して認識精度を上げる(英語混在が頻出するため `en` は常に含める)
- 字幕トークンの扱いは従来どおり: `translation_status: "original"` = 原文、`"translation"` = 翻訳。確定(`is_final: true`)トークンをLiveKitテキストストリームで相手に配信
- 字幕UI: 相手の発話は「翻訳文をメイン、原文を小さく併記」。非確定トークンはグレー表示→確定で通常表示(既存挙動を踏襲)

### 受け入れ基準(EARS風)
- WHEN 患者がドイツ語で話す THEN 医師画面に日本語訳字幕が1〜2秒以内に表示される
- WHEN 患者が文中で英語に切り替える THEN 英語部分も日本語訳されて表示される(翻訳欠落がない)
- WHEN 医師が日本語で話す THEN 患者画面に患者の選択言語の訳が表示される
- WHEN 患者がSoniox対応言語のいずれか(例: 繁体字中国語、ベトナム語)を選択する THEN コード変更なしにその言語で翻訳字幕が機能する(言語リストは一元管理された定数から取得)
- 翻訳トークンにはタイムスタンプが無い前提でUIを実装する(原文トークンとの1対1対応を仮定しない)

> 注(kickoff 検証 2026-07-13): Soniox 公式の Supported languages は60言語で、中国語は `zh` の単一エントリ。簡体/繁体・広東語の区別は存在しない。プログラム上の正は Get models API。

## Task 2: 会話の文字起こしをSupabaseに逐次保存(医師クライアント単独書き込み)

### 目的
診療中の会話(原文+翻訳)をテキストとして永続化し、診療後に医師がカルテへコピペできるようにする。音声録音は不要。サーバーエージェント(LiveKit Agents)も導入しない。

### 設計方針
- **書き込み役は医師クライアントのみ**。医師はClerk認証済みなのでRLSが素直に組める。患者(ゲスト)には書き込ませない
- 医師クライアントには「自分の発話(自セッションのトークン)」と「患者の発話(LiveKitテキストストリーム経由)」の両方が揃っているので、全会話を保存できる
- **逐次保存**: 確定セグメント単位(Sonioxのendpoint検出 or 一定デバウンス2〜5秒)でinsert。セッション終了時の一括保存のみは禁止(タブクローズ/クラッシュで全ロスするため)
- 保険: `pagehide` イベントで `navigator.sendBeacon` により未送信バッファをフラッシュ
- リトライ・再送で重複しないよう、`(session_id, speaker, seq)` のunique制約 + upsert

### テーブル案
```sql
create table consultation_transcripts (
  id uuid primary key default gen_random_uuid(),
  session_id uuid not null references consultations(id),
  speaker text not null,          -- 'doctor' | 'patient'
  language text,                  -- 検出言語 (ja/de/en/...)
  original_text text not null,
  translated_text text,
  seq int not null,               -- クライアント採番の連番
  spoken_at timestamptz not null default now(),
  created_at timestamptz not null default now(),
  unique (session_id, speaker, seq)
);
```
- RLS: 医師(認証ユーザー)が自分の担当するsession_idにのみinsert/select可。既存のconsultationsテーブル構造に合わせて調整
- 既存スキーマに `consultations` 相当が無ければ最小限で新設してよい

### 受け入れ基準
- WHEN 診療中に発話が確定する THEN 数秒以内にSupabaseへ行が追加される
- WHEN ネットワーク瞬断後に再送される THEN 重複行が発生しない(upsertで吸収)
- WHEN 医師がタブを閉じる THEN 直前までの未送信分がsendBeaconで送信される(ベストエフォート)
- IF 患者クライアントからのinsertが試行される THEN RLSで拒否される

## Task 3: 診療後のトランスクリプト表示・コピー画面

### 内容
- 診療終了後(または医師用ダッシュボードから)、session_id単位で会話全文を時系列表示するページ
- 表示フォーマット例:
  ```
  [医師 ja] こんにちは、今日はどうされましたか
  [患者 de→ja] 昨日から喉が痛いです (原文: Ich habe seit gestern Halsschmerzen)
  ```
- 「全文コピー」ボタン(プレーンテキストでクリップボードへ)。カルテシステムへのペーストを想定した、装飾のない整形テキスト
- (任意・余力があれば)全文をLLMに1回投げてSOAP形式の下書きを生成し、それもコピー可能にする。実装する場合はサーバー側(API Route)からLLMを呼ぶこと

### 受け入れ基準
- WHEN 医師がコピー ボタンを押す THEN 会話全文が話者・言語ラベル付きプレーンテキストでクリップボードに入る
- 医師本人が担当した診療のみ閲覧可能(RLS/認可チェック)

## 今回は実装しない(要件・決定として記録のみ)

以下は実装せず、`docs/`(またはバックログ/ADR)に記録だけ残すこと:

1. **同時接続数の監視・事前チェック**(バックログ)
   - Sonioxはプロジェクト/組織単位でリアルタイムWebSocketの同時実行数上限があり、超過時は429 (`limit_exceeded`) が返る。現在値と上限は `GET /v1/concurrency-limits` 相当のAPIで取得可能(要APIキーのためサーバー側=API Route経由で叩く。DBは不要)
   - 将来要件: 診療開始前に空きを確認し、上限到達時はユーザーに待機を案内 + 429時の指数バックオフ再接続。1診療 = 2ストリーム(医師+患者)換算でキャパ計画する
   - 上限引き上げはSoniox Consoleから申請可能(1〜3営業日)という運用メモも残す

2. **サーバー側翻訳(LiveKit Agents)は不採用**(ADR)
   - 決定: 1対1診療・字幕用途ではクライアント直結が遅延・コスト・運用すべてで優位のため採用しない
   - 再検討トリガー: (a) 音声録音の保存が要件化した場合、(b) 3言語以上の参加者(通訳・家族)対応、(c) 通話中のリアルタイム要約が必要になった場合

3. **コンプライアンス注意**(docsに明記)
   - 文字起こし保存を開始した時点で診療情報=要配慮個人情報の保存にあたる。患者向け同意フロー(初回同意)に「会話内容をテキスト記録する」旨を追加する必要がある(UI文言はプレースホルダでよいので同意チェックの導線だけ用意)
   - Sonioxのセッションはpause中も課金されるため、待機時は切断→再接続とする

## 全体の制約
- 既存のクライアント直結アーキテクチャ・一時APIキー発行フローは変更しない
- Soniox恒久APIキーをクライアントに露出させない(既存方針の維持)
- モデルは `stt-rt-v5` を明示指定
- 実装前に、疑問点や既存コードとの不整合があれば着手前に質問すること
