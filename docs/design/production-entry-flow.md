# 本実装の診療入室フロー

## 基本フロー

1. 医師が Clerk でログインする。
2. 医師が診療セッションを作成する。患者言語は予約時、または患者入室時に選択する。
3. システムが患者用リンクを発行し、医師が患者へ共有する。
4. 患者がリンクを開き、対象の診療であることと自分が字幕として受け取りたい言語を確認する。
5. 患者がゲストとして LiveKit room に参加する。

患者向け選択肢は Soniox 対応言語の一元管理されたリストから生成し、言語コードを推測または個別ハードコードしない。検証済み事実は対応60言語、中国語は `zh` の単一エントリである。プログラム上の正は Soniox Get models API とする。

## 患者言語の自動連携案

患者の選択言語を検証したうえで LiveKit participant attributes に載せ、医師クライアントが参加者更新イベントから受け取る。医師側 Soniox セッションの one_way `target_language` をその値へ自動設定する。participant attribute は設定値の伝達にだけ使い、認証・診療アクセス権の根拠にはしない。

患者入室前に言語が未確定なら、医師側 Soniox セッションは文字起こしのみで開始する。患者属性から言語が判明した時点で現在の Soniox 接続を終了し、one_way target を設定して再接続する。患者側は医師向けに target=`ja` とする。恒久 Soniox API key はクライアントへ出さず、既存の一時キー方式と `stt-rt-v5` を維持する。

予約時に患者言語が確定済みなら、診療セッションの値を入室前から使える。予約値と入室時確認値が異なる場合の優先順位は未決事項とする。

## このリポの現行との差分

このリポでは認証・予約・DB・患者リンクがなく、参加者が role を自己申告し、医師がロビーで患者言語を手動選択する。本実装では医師認証と診療セッションを起点にし、患者本人の確認結果を participant attributes 経由で医師セッションへ自動連携する。

字幕 data channel の受信時は、送信者 identity を LiveKit `RoomEvent.DataReceived` イベントの `participant` オブジェクトと必ず突合する。payload が自己申告する identity が実際の送信者と一致しない場合は棄却し、identity と role の決定に payload の自己申告値を信用しない。表示用の identity と role は、`participant` オブジェクトとサーバー側で検証した診療参加コンテキストから導出する。

## 未決事項

- 予約時言語と患者の入室時変更の優先順位、および確定値の保存先。
- 患者リンクの署名形式、有効期限、再入室・再利用の扱い。
- 複数タブや再入室で患者言語属性が競合した場合の採用規則。想定は1対1診療。
- 再接続中の字幕欠落を画面でどう案内するか。
