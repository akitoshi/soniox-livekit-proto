# バックログ: Soniox 同時実行数の事前確認と 429 対応

## 背景

Soniox のリアルタイム WebSocket にはプロジェクト/組織単位の同時実行数上限があり、超過時は HTTP 429 `limit_exceeded` となる。1診療は医師と患者の2つの Soniox ストリームを消費する。

## 将来実装

- Soniox の `GET /v1/concurrency-limits` 相当から現在値と上限を取得する。恒久 API key が必要なため Next.js API Route から呼び、ブラウザへは必要な空き状況だけを返す。DB は不要。
- 診療開始前に最低2ストリーム分の空きを確認する。上限到達時は診療を開始せず、ユーザーへ待機を案内する。
- 事前確認後にも競合し得るため、WebSocket 接続時の 429 を必ず処理する。指数バックオフと jitter を付けて再接続し、待機中・再試行中・中止を UI に示す。
- キャパシティ計画は `同時診療数 × 2ストリーム` を基準にし、運用余裕を別途定める。
- 上限引き上げは Soniox Console から申請し、反映目安を1〜3営業日として運用計画へ織り込む。

API Route は Soniox の生レスポンスをそのまま返さず、許可した数値・状態だけに絞る。恒久 API key をクライアント、ログ、エラーメッセージへ出さない。

## 受け入れ観点

- 空きが2未満なら診療開始前に待機案内が出る。
- 事前確認と実接続の間に上限へ達しても、429 を通常障害と区別して再試行できる。
- 再試行が同時集中しない。ユーザーが待機を中止できる。
- 実測同時診療数とストリーム消費がキャパシティ計画と一致する。
